# AGENT 指南

本文档面向进入本仓库工作的 agent、协作者和维护者。

它的职责不是定义人格或语气，而是说明这个项目是什么、代码放在哪里、应该如何运行，以及在修改仓库时需要遵守哪些约束。角色、语气和高层决策偏好应优先放在 `SOUL.md` 中维护。

## 1. 项目概述

这是一个基于 `pi` 生态的最小演示项目，用来展示以下几件事如何在同一个仓库里协同工作：

- 项目级上下文文件：`SOUL.md` 与 `AGENT.md`
- 项目本地 skills：`.pi/skills/<skill-name>/SKILL.md`
- 通过 SDK 注册的自定义工具
- 共享同一套项目上下文的 CLI chat 与 Web UI

这个仓库的重点不是做一个完整产品，而是演示“项目上下文 + 本地 skill + 工具 + 前端入口”这套机制如何落地。因此，任何改动都应优先服务于演示清晰度、可读性和可扩展性。

## 2. 核心目录与文件

进入仓库后，优先理解下面这些文件：

- `README.md`
  说明项目用途、安装方法、运行命令和示例用法。
- `SOUL.md`
  定义 agent 的角色、语气、身份和决策优先级。
- `AGENT.md`
  定义项目说明、运行规则、协作约束和修改原则。
- `src/index.ts`
  CLI 会话入口，负责加载项目上下文、注册自定义工具、创建 agent session。
- `src/catalog.ts`
  本地示例数据与 `search_journals` 的检索逻辑。
- `web/main.ts`
  Web UI 入口，负责组装系统提示词、管理 slash command 和前端 agent 行为。
- `.pi/skills/`
  项目本地 skills。每个 skill 目录至少包含一个 `SKILL.md`，会被项目运行时发现，并在 Web UI 中以 `/<skill-name>` 的形式暴露。
- `.pi/extensions/`
  项目级扩展与 provider 配置。

## 3. 运行方式

常用命令如下：

- `npm install`
  安装依赖。
- `npm run chat`
  启动命令行交互式对话。
- `npm run demo`
  运行一个内置的一次性演示 prompt。
- `npm run web:dev`
  启动 Web UI 开发服务器。
- `npm run check`
  运行 TypeScript 类型检查。
- `npm run web:check`
  运行 Web 相关的 TypeScript 类型检查。

本项目依赖模型提供方的 API key。默认示例使用本地 `.env` 或 `~/.pi/agent/auth.json` 中的凭据。

## 4. 文档职责边界

为了避免上下文混乱，请遵守以下分工：

- `SOUL.md` 负责回答“你是谁、用什么语气、优先考虑什么”。
- `AGENT.md` 负责回答“这个仓库是什么、如何工作、有哪些约束”。
- `SKILL.md` 负责回答“当某个 skill 被激活时，当前任务应如何执行”。

不要把三类信息混写在一起。

特别是：

- 不要把项目目录说明、运行命令塞进 `SOUL.md`
- 不要把人格设定、口吻要求塞进 `AGENT.md`
- 不要把通用项目规则硬编码进某个单独的 skill

## 5. Agent 进入仓库后的默认工作顺序

当一个 agent 第一次进入本仓库时，默认应按以下顺序建立上下文：

1. 阅读 `README.md`，确认项目目标和运行方式。
2. 阅读 `SOUL.md`，确认角色与表达风格。
3. 阅读 `AGENT.md`，确认项目约束与协作边界。
4. 如用户请求明确匹配某个 skill，再读取对应 `.pi/skills/<skill-name>/SKILL.md`。
5. 需要理解实现细节时，再进入 `src/`、`web/` 或 `.pi/extensions/` 查阅代码。

如果用户请求与某个项目本地 skill 明确匹配，应优先遵循该 skill 的流程；如果没有激活 skill，则按 `SOUL.md + AGENT.md` 的组合执行。

## 6. Skill 与工具使用约束

### 项目本地 Skill

- 本地 skills 放在 `.pi/skills/`。
- 每个 skill 都应是可独立理解、可独立启用的最小单元。
- 新增 skill 时，优先保持目录清晰、命名直观、说明完整。
- skill 的内容应聚焦任务流程，不要重复复制整份项目说明。

### Slash Command 规则

- Web UI 中的本地 skill 通过 `/<skill-name>` 激活。
- 未激活 skill 时，不应假设存在“隐藏的默认 skill”。
- `/clear` 用于退出当前 active skill，恢复到普通对话。

### 自定义工具规则

当前项目包含示例工具 `search_journals`。

使用约束如下：

- 只有在“期刊推荐、投稿选择、投稿策略”相关任务中才应调用 `search_journals`。
- 工具输出基于本地 demo catalog，只能作为示例数据使用。
- 不要把该工具包装成真实完整的期刊数据库。
- 如果本地 catalog 无匹配结果，应明确说明“这是 demo 数据限制”，而不是编造结果。

## 7. 修改代码时的原则

这是一个示例仓库，不是大型生产系统。修改时应优先遵守以下原则：

- 优先保证演示路径清晰，而不是功能堆积。
- 优先保持实现简单、直接、便于阅读。
- 能复用现有模式时，不要平行造一套新机制。
- 新增能力时，尽量同时考虑 CLI 与 Web UI 是否应共享同一套上下文或规则。
- 不要为了“更通用”而引入与当前 demo 无关的复杂抽象。

如果修改涉及 `SOUL.md`、`AGENT.md`、`.pi/skills/` 或 `web/main.ts`，要特别检查它们之间是否仍然保持一致：

- `README.md` 中描述的行为，代码里应该能跑通。
- `SOUL.md` 中定义的角色，不应与 `AGENT.md` 冲突。
- `AGENT.md` 中写明的规则，不应与 Web UI 或 CLI 的实际装配逻辑冲突。
- skill 的启用方式，应与 `web/main.ts` 中的 slash command 逻辑一致。

## 8. 对外输出时的基本要求

在这个 demo 项目里，agent 的输出通常应满足以下要求：

- 简洁
- 明确
- 可执行
- 与当前项目上下文一致

除非用户明确要求，否则不要：

- 编造仓库中不存在的功能
- 假装接入了外部数据库或线上服务
- 声称某个 skill 会自动生效，如果当前并未激活它
- 把 demo 数据说成真实世界的完整知识源

## 9. 适合放进本文件的内容

下面这些内容适合写进 `AGENT.md`：

- 仓库用途
- 关键目录说明
- 运行入口
- 协作约束
- 工具与 skill 的使用边界
- 修改代码时的项目级原则

下面这些内容不适合写进 `AGENT.md`：

- 纯人格设定
- 纯语气要求
- 某个单一任务的详细执行步骤
- 与当前仓库无关的通用大而全规范

## 10. 维护建议

当项目演示重点发生变化时，应优先同步更新以下文件：

- `README.md`
- `SOUL.md`
- `AGENT.md`
- `.pi/skills/` 下相关 `SKILL.md`

如果这几处内容不一致，优先修正文档，使其重新反映代码中的真实行为，而不是依赖隐含约定。
